# 05-设计-交互系统设计

## 设计框架

* 定义高层次的屏幕布局
* 定义产品的工作流、行为、组织
* 忽略细节

### 步骤

1. 定义外形因素和输入方法
   * 外形因素：结合产品的特点和约束（人物角色、场景剧本）
   * 产品输入方法：产品和用户互动的形式，取决于产品的外形和人物角色的能力和喜好
2. 定义功能和数据元素
   * 数据元素：交互产品的基本主题（相片、电子邮件、订单）
   * 功能元素：对数据元素的操作，输入/放置数据元素的位置
3. 决定功能组合层次
   * 元素分组
4. 勾画大致的设计框架：方块图
5. 构建关键情景场景剧本：故事板
   * 关键情景：用户最频繁使用的主要路径
6. 通过验证性的场景剧本检查设计：考虑更多的情景
   * 关键线路变种场景剧本：关键途径的替代（不打电话，而是发邮件）
   * 必须使用的场景剧本：必须要被执行，但不经常发生（卖二手手机清除个人信息）
   * 边缘情形使用场景剧本：非典型产品具备，但不太常用的功能（添加同名联系人）

## 情绪板

* 用于产品视觉设计：色彩、图形、质感、构图、字体
* 由能代表用户情绪的文本、元素、图片拼接而成的客观表达设计理念
* 步骤
  * 寻找主题相关的关键词
  * 关键词联想
  * 搜索关键词图片
  * 创建情绪板
  * 视觉设计

## 设计中的折衷

* 个性化 & 配置
  * 个性化：必须简单易用、提供预览、可撤销
  * 配置：移动、添加或者删除持久对象
* 本地化 & 国际化
  * 本地化：针对特定地区，进行翻译、增添特色
  * 国际化：设计时考虑移植到不同地区，将软件和特定语言脱钩
* 审美学 & 实用性：交互中先实现良好的布局，再增加美学效果
  * 留白
  * 对齐
  * 避免滚动
  * 避免花里胡哨的背景

## 软件设计中的考虑

* 文字提示
  * 积极的文字表达
  * 消除歧义
  * 让软件友好、体贴
  * 有趣的用语
* “聪明”的软件：避免反人类设计
* 避免邀功的对话框
* 加快响应时间
* 减轻用户记忆负担
  * 用户需记住软件操作方法 + 任务相关的领域知识
  * 好的做法：记忆用户之前的操作、选择
* 减少用户等待感
  * 进度条
  * 动画
  * 占位符/骨架屏
  * 分配任务，分散用户注意力
  * 降低用户期望值
* 设计好的出错信息
  * 语言清晰，不暴露代码
  * 精炼正确
  * 对解决问题提供建设性帮助
  * 友好，不威胁责备用户

## 简化设计的策略

### 删除

* 删除杂乱的特性
  * 设计师：专注于解决重要问题
  * 用户：专注于完成自己目标
  * 避免得到简单功能叠加的毫无特色的产品
  * 保证交付真正有价值的功能、内容
* 要点
  * 关注核心：基本功能、影响日常体验的功能
  * 砍掉残缺功能：放弃沉没成本
  * 不盲从用户的要求
* 不要删减过多
  * 启发式原则：用户享有控制权和自主权

#### 方法

* 删除错误：合并选择框，使用输入框替代下拉菜单
* 删除视觉混乱：集中注意力在重要的内容上
  * 使用空白或轻微背景来划分页面，不要使用线条
  * 少用强调，仅加粗
  * 避免使用粗黑线，匀称、浅色的线更好
  * 控制信息的层次，标题、子标题、正文
  * 减少元素大小、形状的变化
* 删减文字
  * 引荐性文字：欢迎
  * 不必要的说明/繁琐的解释
  * 使用描述性链接：“更多内容”
  * 精简句子

### 组织

* 分块、分类
  * 7±2法则
* 分类标准
  * 名词：字母表、时间、空间顺序
  * 动词：围绕行为组织
* 确定清晰的分类标准
  * 启发式原则：依赖识别而非记忆

#### 方法

* 利用不可见的网格对齐元素
* 重要的元素变大
* 相似的图标放在一起
* 感知分层：区分不同层级的信息
* 期望路径：用户期望的浏览路径不一定和设计路径一致

### 隐藏

* 低成本的简化方案
  * 用户不会因为不常用的功能分散注意力
  * 删除不必要功能的开始
* 隐藏内容
  * 主流用户很少使用，自身需要更新的功能
  * 细节
  * 选项/偏好
  * 特定于地区的信息：时间、日期

#### 自定义

* 主流用户：提供简单的自定义工具
  * 展示个性，更换主题、头像
  * 不应让用户自定义软件的界面
* 高级用户：渐进展示
  * 折叠拓展功能

#### 要点

* 适时出现：在合适的时机显示被隐藏的功能，无需过分强调
* 让功能易于发现
  * 给隐藏功能打上标签：“更多”、“高级”
  * 标签应在用户的关注点之内

### 转移

#### 在设备之间转移

| 维度     | 移动平台          | 桌面平台/笔记本         |
| ------ | ------------- | ---------------- |
| 拍摄能力   | 可以拍摄任何景物      | 只能拍到用户(通过网络摄像头)  |
| 文本输入   | 输入少量文本        | 输入大量文本           |
| 数据传输速度 | 很难加快数据传输速度    | 能适当加快数据传输速度      |
| 信息显示   | 显示少量信息        | 显示大量信息           |
| 信息保存   | 保存适量信息        | 保存大量信息           |
| 使用场景   | 随时随地使用        | 只能坐下来使用          |
| 位置识别   | 能够精确识别位置和方向   | 难以识别位置和方向        |
| 连接方式   | 通过无线网络连接到其他设备 | 通过有线和无线网络连接到其他设备 |

#### 向用户转移

* 将复杂的工作留给用户
* 让用户有更多自主权

| 人擅长          | 计算机擅长    |
| ------------ | -------- |
| 设定目标和制定规划    | 执行程序     |
| 估算           | 精确计算     |
| 辨别信息         | 存储和检索信息  |
| 做图表          | 复制       |
| 在包含少数项的列表中选择 | 对大型列表排序  |
| 做预算          | 度量       |
| 想象           | 交叉引用详细信息 |

### 组合策略

* 删除不必要的
* 组织要提供的
* 隐藏非核心的
* 转移

## 卡片分类

* 设计挑战：如何让用户在产品中快速找到他们需要的信息
* 卡片分类法：一种通过让参与者对代表信息元素的卡片进行分类，以理解用户心智模型的研究方法
* 焦点：从用户视角理解问题域的概念、术语、关系

### 优点

* 代价小，低成本
* 较可用性测试更高效
* 研究方式灵活：
  * 定性研究：参与者数量少，采用一对一的实体卡片形式
  * 定量研究：通常需要至少15名参与者，采用一对多的实体卡片或在线分类
* 应用场景
  * 导航研究：信息架构、导航设计
  * 界面布局研究：哪些功能控件应放在一起

### 具体方法

#### 卡片内容

* 导航研究：内容标题、活动/任务名称、任何可能出现在页面上的内容
* 表格/页面设计：数据项或界面元素
* 领域研究：问题域中的核心概念、任务或目标

#### 术语

* 来源于用户
  * 现场研究
  * 探索性卡片分类：不预先定义类别，让参与者自由分类
* 项目名称
  * 简短/常用
  * 无歧义
  * 避免不明的内部品牌名称

#### 分组

* 避免
  * 抽象/模糊的组名：工具、服务
  * 包罗万象的组名：通用、其他、杂项
* 把通用类别放在最后
* 组名应互斥、独立
