## C++ 历史

> C++ 设计理念、演化历程（包括主要的贡献者），Simula 67在其中的作用。

* 设计理念
  * 效率
  * 实用性优于艺术性严谨性
  * 相信程序员：允许一个有用的特征比防止各种错误使用更重要
* 演化历程：C & Simula 67 -> C with Classes - > C++
* Simula 67 的作用：引入了类、对象、虚函数等机制，对 C++ 的面向对象特性有重要影响

> C 和 C++ 的关系、异同、C 与 C++ 混合编程时要注意的问题

* 关系：C++ 是 C 的超集
* 相同点：
  * C++ 支持 C 所支持的全部编程技巧
  * 任何 C 程序都能被 C++ 用基本相同的方法编写，并具备同等开销
* 不同点：C++ 增加类、继承、模板等高级特性
* C与Ｃ++混合编程时要注意的问题
  * 避免使用 C++ 关键字作为变量名
  * 使用 `extern "C"` 声明 C 函数，防止 name mangling
  * 统一内存管理方式
  * 避免抛出异常

## 基础语法

> C++ 头文件中存放的内容

* 存放函数/全局变量声明，编译时不会生成代码/占用存储空间
* 常量定义：`inline constexpr` 或 `const`，避免重复定义
* 类型定义：`struct`、`class`、`enum`、`typedef`、`using`
* **模板的声明 + 实现**
* `inline` 函数定义

> `static` 关键字的作用

* 全局作用域下：限制变量或函数的可见性为当前文件
* 类内：声明静态成员变量或函数，属于类而非对象实例
  * `static` 变量需要在类外初始化
  * `inline static` 变量可在类内初始化（C++17 起支持）
  * `const static` 整型或枚举类型变量可在类内初始化
  * `const static` 非整型变量需在类外初始化
* 函数内：变量只初始化一次，保持其值跨函数调用

> 影响表达式求值的因素。表达式是否存在副作用？

* 影响求值的因素：操作符、操作数、优先级、结合性、短路求值、求值顺序、类型转换
* 表达式可能存在副作用：修改变量、调用函数、I/O 操作、内存操作、抛出异常……

> `inline` 函数与宏的区别？`inline` 函数的优点和缺点？

* 定义：编译时，编译器将内联函数体直接展开，插入调用处
* 区别
  * 宏：预处理器文本替换，无类型检查，无作用域限制，调试困难
  * 内联函数：编译器处理，有类型检查，避免出错，可调试
* 优点：减小函数调用开销、类型安全
* 缺点：增加代码体积，可能导致缓存效率降低、不保证一定内联
* `inline` 函数体必须在头文件中：以便每个使用该函数的编译单元都能看到其定义，否则无法内联

> `const` 的几种使用方法

* 常量变量：变量无法修改
* 指针
  * `const int *p`：指向常量整数的指针，不能修改 `*p` 指向的整数，可修改 `p` 指向的位置
  * `int *const p`：常量指针，不能修改 `p` 指向的位置，可修改 `*p` 指向的整数
* 成员变量：类的成员变量不可修改
* 成员函数：函数不会修改对象状态，可调用 `const` 成员函数的对象也必须是 `const`（`const` 函数可修改 `mutable` 成员变量）
* 函数
  * 参数：函数内不能修改传入参数
  * 返回值：返回常量，防止调用者修改

> `constexpr` 的作用？ `const` `constexpr` `consteval` 有何不同？

| 关键点   | `const` | `constexpr`                 | `consteval`   |
| ----- | ------- | --------------------------- | ------------- |
| 是否只读  | 是       | 是                           | 是             |
| 作用    | 定义只读常量  | 定义“可用于常量表达式”的常量或函数，希望在编译期求值 | 定义必须在编译期求值的函数 |
| 编译期求值 | 否       | 不强制                         | 强制            |
| 运行期求值 | 是       | 视情况                         | 否             |

* 被 `constexpr` 和 `consteval` 修饰的函数不得包含任何有运行时行为的代码：I/O 操作、动态内存分配、异常处理等

## 类型

> `C++` 中隐式类型转换和显式（强制）类型转换的区别？

| 特性   | 隐式类型转换            | 显式类型转换       |
| ---- | ----------------- | ------------ |
| 触发方式 | 编译器自动，无需编写代码      | 程序员手动        |
| 适用范围 | 兼容类型（基本类型、派生类-基类） | 任意类型（只要语义允许） |
| 安全性  | 存在风险，可能会精度丢失/溢出   | 意图明确，更可读，更安全 |

> C++ 中的各类显式类型转换

| 转换类型                                 | 适用场景                       | 说明                      |
| ------------------------------------ | -------------------------- | ----------------------- |
| C 风格转换 `(type)expression`            | 避免使用                       | 不安全                     |
| `static_cast<type>(expression)`      | 常规转换                       | 类型安全，编译时检查              |
| `const_cast<type>(expression)`       | 添加/移除 `const` 或 `volatile` | 仅修改类型修饰符，返回值仍为常量，修改会 UB |
| `reinterpret_cast<type>(expression)` | 转换指针类型                     | 底层，危险                   |
| `dynamic_cast<type>(expression)`     | 基类与派生类指针转换                 | 运行时检查                   |
| `any_cast<type>(expression)`         | 从 `std::any` 提取类型          | 类型安全，运行时检查              |

> `static_cast` 的使用场景

* 算术类型转换
* 枚举 <-> 整型
* `void*` <-> 对象指针
* 派生类 -> 基类（upcast）
* 基类 -> 派生类（downcast）：不安全，无运行时检查，使用 `dynamic_cast` 更安全
* 当类定义了`operator T()`时，调用转换方法

> `any_cast` 和 `dynamic_cast` 在转换失败时的表现

* `expression` 为指针类型：返回 `nullptr`
* `expression` 为引用类型：抛出 `std::bad_any_cast` / `std::bad_cast` 异常

> `union` 与 `struct`、`class` 的区别

* `union`：所有成员共享同一块内存
* `struct` 和 `class`：每个成员有独立内存

> `std::any` `std::variant` `std::optional` `std::tuple` 分别的作用和使用场景

* `std::any`：存储任意**可拷贝**类型的单一值，适用于需要存储不同类型但不确定类型的场景
* `std::variant`：存储多种类型中的一种，适用于需要在几种已知类型中切换的场景
* `std::optional`：表示可能存在或不存在的值，适用于函数返回值可能为空的场景
* `std::tuple`：存储多个不同类型的值，适用于函数返回多个值的场景

## 指针 & 引用 & 内存管理

> C++ 中数组和指针的区别？如何⽤指针操作数组元素？

* 区别：
  * 类型&长度信息：数组携带长度信息 `type[length]`，指针不携带 `type*`
  * `sizeof` 结果：数组返回整个数组大小，指针返回指针大小
  * 可修改性：数组是不可修改的左值，不可赋值，指针可赋值
  * 退化：数组作为右值/函数参数（除了 `sizeof` `&` `decltype` 之外的情况）会退化为指针，指针不会退化
* 用指针操作数组元素：解引用指针，对指针进行加减运算：`(p + i)` 指向第 `i` 个元素，偏移 `i * sizeof(type)` 字节

> 为何在 C++ 中推荐使用 `nullptr` 而非 `NULL`？

* `NULL` 通常定义为整数 `0`，在**重载函数**时可能引起歧义
* `nullptr` 是专门的空指针类型，避免歧义，更安全

> 简述 `C++` 的内存管理机制。栈区、堆区、静态存储区的区别？

| 内存区域  | 管理方式                   | 存储内容           | 生命周期        |
| ----- | ---------------------- | -------------- | ----------- |
| 栈区    | 编译器自动管理                | 局部变量、函数参数、返回地址 | 随作用域自动创建/销毁 |
| 堆区    | 程序员手动管理                | 动态分配对象         | 程序员控制       |
| 静态存储区 | 静态，编译时确定，程序启动时分配，结束时释放 | 全局变量、静态变量、常量   | 程序整个生命周期    |
| 文字常量区 | 静态，只读                  | 字符串字面量等只读数据    | 程序整个生命周期    |

> 在 C++ 中，什么是 RAII？举出例⼦，这种约定在保证异常安全性⽅⾯的作⽤？

* RAII：Resource Acquisition Is Initialization
* 资源的获取与对象的生命周期绑定
  * 在构造函数中获取资源
  * 在析构函数中释放资源
  * 利用 C++ 对象生命周期自动管理资源
* 析构函数离开作用域自动调用，**即使发生异常也能确保资源释放**，保证异常安全性。

```cpp
{
  std::unique_ptr<Resource> res(new Resource());
  res->use();
  throw std::runtime_error("error"); // 即使抛出异常，资源也会被释放
} // 离开作用域时，res 自动析构，资源被释放
```

> C++ 中常见的智能指针有哪些？它们各自的特点和适用场景是什么？

| 智能指针         | 特点                                     | 适用场景                            | 关键点                                              |
| ------------ | -------------------------------------- | ------------------------------- | ------------------------------------------------ |
| `unique_ptr` | 独占所有权，不可拷贝、可移动；作用域结束自动释放资源             | 明确的单一所有者对象；RAII 资源管理            | 转移所有权：`std::unique_ptr<int> p2 = std::move(p1);` |
| `shared_ptr` | 共享所有权，基于引用计数；最后一个 `shared_ptr` 析构时释放资源 | 对象被多个模块共享，生命周期难以确定              | 引用计数带来额外开销；不当使用可能造成循环引用                          |
| `weak_ptr`   | 对 `shared_ptr` 所管理对象的非拥有性观察；不增加引用计数    | 解决 `shared_ptr` 循环引用问题；缓存、观察者模式 | 使用前必须通过 `lock()` 获取 `shared_ptr` 并检查对象是否仍然存在     |

> 引用和指针的区别

* 引用：变量的别名，必须初始化（非空），不可修改引用目标
* 指针：变量存储另一个变量的地址，可为空，可修改指向的地址

## 函数

> 值传递、引用传递、指针传递的区别

* 值传递：把值的副本传递给被调函数，函数内改动不影响原值
* 引用传递：把变量的引用传递给被调函数，函数内改动影响原值，相较于指针更安全
* 指针传递：把变量的地址传递给被调函数，函数内通过指针改动影响原值，指针可能为空

> 何时应将返回值设为引用类型？可能存在什么风险？

* 适用场景
  * 需要修改返回的对象/类内部成员
  * 避免拷贝开销
  * 链式调用
* 风险
  * 返回临时对象/局部变量的引用：悬空引用
  * 返回常量的引用：修改产生 UB

> 函数调用的约定

| 调用约定               | 参数压栈顺序                             | 谁负责清栈                        | 常见用途                      |
| ------------------ | ---------------------------------- | ---------------------------- | ------------------------- |
| cdecl              | 从右到左                               | caller                       | C/C++ 默认；支持可变参数（`printf`） |
| stdcall            | 从右到左                               | callee                       | Windows API 常用；不支持可变参数    |
| fastcall           | 前两个参数进寄存器（通常 `ECX`, `EDX`），剩下右到左入栈 | callee                       | 编译器优化、某些系统函数              |
| thiscall（C++ 成员函数） | `this` 放 `ECX`，其它参数右到左             | Visual C++ 下 callee（gcc 有变化） | C++ 成员函数默认约定              |

> Lambda 函数与函数对象的异同？

* Lambda 函数和函数对象都是重载了 `operator()` 的可调用对象

| 特性       | Lambda 函数                          | 函数对象         |
| -------- | ---------------------------------- | ------------ |
| **定义方式** | 语法简洁，直接写在代码里                       | 需要显式定义类/结构体  |
| **名字**   | 匿名类型（没有显式类型名，但编译器生成一个唯一类型）         | 有明确类型        |
| **捕获能力** | 可以捕获外部变量（按值或按引用）                   | 不能捕获外部变量     |
| **状态存储** | 无状态存储                              | 可以通过成员变量存储状态 |
| **可重用性** | 匿名，除非用 `auto` 或 `std::function` 保存 | 可多次实例化       |

> 函数指针的缺点？

* 可读性差：语法复杂，难以理解
* 类型安全性差：容易传递错误类型的函数指针
* 灵活性差：不支持捕获外部变量
* 效率低：无法内联，调用开销较大

> 什么是类型擦除？ `std::function` 的作用？可保存哪些类型的可调用对象？

* 类型擦除：在编译期保存具体类型信息，但对外隐藏其具体类型，仅通过统一的接口进行访问，从而使不同类型但接口一致的可调用对象在运行期表现为同一种类型
  * 可调用对象即使参数类型和返回类型完全相同，具体的类型也是不同的
* `std::function`：通用的可调用对象封装器，通过保存不同类型的可调用对象，实现运行时多态
* 可保存的可调用对象类型：普通函数、函数指针、Lambda 表达式、函数对象、`std::bind` 生成的可调用对象等

## 模板

> 什么是模板？模板与函数重载相比解决了什么问题？

* 模板：将类型作为参数进行泛型编程
* 解决的问题：无需针对每一种类型编写重复代码

> 模板的实例化过程是怎样的？什么是按需实例化？

* 在编译时，编译器根据模板参数生成具体类型的代码
* 按需实例化：只有使用模板时才生成代码，避免代码膨胀

> 为什么模板的声明和定义通常必须放在同一个头文件中？请从编译与链接角度说明原因。

* 模板的定义无法单独编译和链接，必须在编译时可见，根据实际用到的类型进行实例化，因此需放在头文件中

> 若模板参数类型不支持某些操作，错误是在何时被发现？为什么？

* 编译时，无法生成对应类型的代码时会报错
* 原因：编译器本身不检查模板代码，只有在实例化时才会检查

> 什么是模板的显式具体化（特化）？它解决了什么问题？

* 针对一些特殊类型，生成和通用模板不同的实现
* 解决的问题：为特定类型提供优化或不同的行为

> 如果模板在编译期就能实现多态，为什么 C++ 仍然需要虚函数机制？

* 模板多态是编译时多态，虚函数是运行时多态
* 模板多态无法处理运行时类型变化的场景
* 虚函数支持动态绑定，适用于需要在运行时决定调用哪个函数的场景

> `define` 替换和 `template` 的区别

* `#define`：预处理器宏，预处理阶段文本替换，易出错
* `template`：编译器处理，类型安全，受作用域控制，易于调试，支持重载/特化，更安全，可读性更高

> 使用模板元编程进行计算与普通运行期计算相比，有何优缺点？

* 优点
  * 编译期计算，运行时无开销
  * 可用于生成复杂类型和代码结构
* 缺点：
  * 编译时间增加
  * 错误信息复杂，难以调试
  * 可读性差，维护困难

> `if constexpr` 与普通 `if` 的本质区别是什么？

* 编译时条件判断，条件为 `true` 分支会被编译，`false` 分支会被忽略

## 异常

> 当异常未被任何 `catch` 捕获时，程序将发生什么？为什么要设计成这种行为？

* 调用 `std::terminate()`，立即终止程序
* 设计原因：防止异常传播到未知状态，避免未定义行为，确保程序安全终止

> 异常对象在抛出和捕获过程中是如何传递的？是否会发生拷贝？

* 拷贝异常对象到异常处理机制中，若捕获时使用引用则避免拷贝

> `throw;` 与 `throw e;` 有何区别？在什么情况下必须使用前者？

* `throw;`：在 `catch` 块内重新抛出当前异常
* `throw e;`：抛出新的异常对象 `e`

> 如果一个类在析构函数中抛出异常，会产生什么后果？为什么应避免这种设计？

* 若已有异常在传播，析构函数抛出异常会调用 `std::terminate()`，导致程序终止

## 构造 & 析构

> 构造函数和其他的成员函数有何不同？

* 构造函数没有返回类型，名称与类名相同
* 自动调用，初始化对象
* 不能被继承
* 不能声明为 `virtual`

> 编译器自动生成的默认构造函数对内置类型成员和类类型成员的初始化行为有何不同？

* 内置类型成员：不初始化，值未定义
* 类类型成员：调用默认构造函数进行初始化

> 在哪些情况下编译器不会为类生成默认构造函数？

* 类定义了其他构造函数
* 类包含引用成员或 `const` 成员且未显式初始化
* `delete` 了默认构造函数

> 将构造函数声明为 `explicit` 的作用是什么？举例说明

* 防止隐式类型转换，避免意外调用构造函数

```cpp
class A {
  public:
  explicit A(int x) {}
};

void f() {
  A a1(10); // 正确
  A a2 = 10; // 错误，隐式转换被禁止
}
```

> 成员初始化列表的执行时机是什么？哪些成员必须在成员初始化列表中初始化？有什么优点？

* 执行时机：在构造函数体执行前
* 必须在初始化列表中初始化的成员：**`const` 成员、引用成员、没有默认构造函数的类类型成员**
* 优点：提高性能，避免默认构造和赋值操作，确保正确初始化

> 常见的初始化方式有哪几种？它们各自的优缺点是什么？

* 括号初始化 `Type var(args);`：语法简洁
* 括号列表初始化 `Type var{args};`：防止窄化转换
* 赋值初始化 `Type var = value;`：语法直观，但可能引起隐式转换
* 显式调用构造函数 `Type var = Type(args);`：冗长，C++17 后不会再生成临时对象

> 请阐述在面向对象程序设计语言中引入构造函数机制的原因

* 自动初始化对象，确保对象在使用前处于有效状态
* 提供统一的初始化接口，简化对象创建过程
* RAII 原则的基础，在构造函数中获取资源
* 便于编译器优化对象创建过程

> 给出控制一个类创建实例个数的手段（举例说明）。

* 将构造函数设为 `private` 或 `protected`，通过成员函数控制实例创建
* 使用 `static` 成员变量记录实例个数

> 如何利用析构函数防止内存漏洞？

* RAII 原则：在析构函数中释放资源，确保资源在对象生命周期结束时被释放

> 什么是析构函数？析构函数的调用时机有哪些？

* 析构函数：类的特殊成员函数，用于清理对象资源
* 调用时机：
  * 栈上对象离开作用域
  * 堆上对象被 `delete`
  * 全局/静态对象程序结束时

> 将析构函数声明为 private 有什么作用？为什么 private 析构函数会导致该类的对象不能定义在栈上？

* 禁止对象在栈上创建
* 原因：栈上对象离开作用域时，编译器无法访问 private 析构函数，导致编译错误

> 继承层次中，为什么基类析构函数是虚函数？

* 在运行时，根据对象的实际类型调用正确的析构函数
* 避免调用基类的析构函数，导致派生类独有的资源泄漏

> `new` `delete` 与 `malloc` `free` 的区别？

* `new`/`delete`：调用构造函数/析构函数，类型安全
* `malloc`/`free`：仅分配/释放内存，不调用构造/析构函数，返回 `void*`，需手动类型转换

### 拷贝/移动构造 拷贝/移动赋值

> 拷贝构造函数与移动构造函数的区别？何时调用拷贝构造函数？

| 操作   | 签名                                            | 调用时机          |
| ---- | --------------------------------------------- | ------------- |
| 拷贝构造 | `ClassName(const ClassName &)`                | 用左值初始化新对象     |
| 移动构造 | `ClassName(ClassName &&) noexcept`            | 用右值初始化新对象     |
| 拷贝赋值 | `ClassName& operator=(const ClassName &)`     | 将一个左值赋值给已存在对象 |
| 移动赋值 | `ClassName& operator=(ClassName &&) noexcept` | 将一个右值赋值给已存在对象 |

> 为什么 C++ 要区分拷贝和移动？

* 拷贝：需要保持源对象不变
* 移动：由于无需保持源对象，允许资源转移，提高性能

> 为什么默认生成的拷贝构造函数可能导致严重的内存问题？

* 默认拷贝构造函数执行浅拷贝，对于指针成员，多个对象共享同一内存，导致双重释放等问题

> 为什么移动构造函数通常应当声明为 `noexcept`？

* 保证在异常安全的上下文中使用
* 如 STL 容器在扩容时移动元素可能抛异常，容器会退化为拷贝构造，影响性能

> 什么是左值，右值？什么是左值引用，常量引用，右值引用？

* 左值 lvalue：有名称，可寻址的对象，生命周期至少到作用域结束
* 纯右值 prvalue：无名称，临时对象，生命周期通常很短
* 将亡值 xvalue：即将被销毁的对象，通常是右值引用类型

| 引用   | 类型签名           | 可绑定对象 | 说明                    |
| ---- | -------------- | ----- | --------------------- |
| 左值引用 | `Type &`       | 左值    |                       |
| 常量引用 | `const Type &` | 左值和右值 | 绑定纯右值时，延长生命周期至作用域结束   |
| 右值引用 | `Type &&`      | 右值    | 实现移动语义，表示指向的对象可安全地被消费 |

> `std::move` 的作用是什么？`std::forward<T>` 的作用是什么？

* `std::move`：将左值转换为右值引用，表示对象的资源可以被“移走”
* `std::forward<T>`：实现完美转发，根据传入的模板参数的类型（左值或右值）保持其值类别，常用于模板函数中将参数传递给其他函数

> 什么是 Rule of Five？什么是 Rule of Zero？

* Rule of Five：如果类定义了以下任意一个，则通常需要定义全部五个
  * 拷贝构造函数
  * 拷贝赋值运算符
  * 移动构造函数
  * 移动赋值运算符
  * 析构函数
* Rule of Zero：提倡依赖智能指针等 RAII 类型来管理资源，如果类无需手动管理内存、文件等资源，则不需要定义上述五个函数，依赖编译器生成的默认实现

> C++ 编译系统赋予一个空类，如：`class Empty { }`，哪些成员函数？

* `ClassName()`：默认构造函数
* `ClassName(const ClassName &)`：拷贝构造函数
* `ClassName(ClassName &&)`：移动构造函数（C++11 起）
* `~ClassName()`：析构函数
* `ClassName &operator=(const ClassName &)`：拷贝赋值函数
* `ClassName &operator=(ClassName &&)`：移动赋值函数（C++11 起）
* `ClassName* operator&()`：取地址操作符
* `const ClassName* operator&() const`：常量取地址操作符

## 友元

> 简述友元的分类以及作用，以及常见的使用误区

* 作用：允许非成员函数或其他类访问类的私有和保护成员
* 分类
  * 友元函数：`friend ret func(params);`
  * 友元类：`friend class ClassName;`
  * 友元成员函数：`friend ret ClassName::func(params);`
  * 友元模板：`template<typename T> friend class ClassName;`
* 误区：友元的友元不是友元，友元的基类/派生类不是友元

## 继承

> 简述 C++ 类的权限修饰符。

| 权限   | `public` | `protected` | `private` |
| ---- | -------- | ----------- | --------- |
| 同一个类 | 可访问      | 可访问         | 可访问       |
| 派生类  | 可访问      | 可访问         | 不可访问      |
| 类外部  | 可访问      | 不可访问        | 不可访问      |

> 请简述 C++ 继承类型，并说明`private`继承的存在价值以及为何要谨慎使用`private`继承。

| 继承方式        | 基类的 `public` 成员 | 基类的 `protected` 成员 | 基类的 `private` 成员 |
| ----------- | --------------- | ------------------ | ---------------- |
| `public`    | `public`        | `protected`        | 不可访问             |
| `protected` | `protected`     | `protected`        | 不可访问             |
| `private`   | `private`       | `private`          | 不可访问             |

* 以上所述权限是对于**类外部的访问控制**，派生类本身还是可以访问基类的 `public` 和 `protected` 成员
* `public` 继承需符合 is-a 关系
* `private` 继承在实现层面复用代码，同时不希望对用户暴露基类的接口
* 谨慎使用：增加耦合，导致代码可读性和可维护性降低

> 什么是“对象切片”？给出实例，原因及解决方法？

* 派生类中额外的成员在内存中，排布在基类成员的后面
* 当派生类按值赋值给基类对象时，基类对象的`size`以外的部分会被切掉，导致数据丢失
* 实例：将 `Derived` 对象赋值给 `Base[]` 数组
* 解决方法：使用指针或引用

> 若派生类未在初始化列表中显式指定基类构造函数，将发生什么？

* 编译器会调用基类的默认构造函数进行初始化
* 若无默认构造函数，编译错误

### 多继承 & 虚继承

> 多继承的语法是什么？当多个基类中存在同名成员时，程序员应如何显式消除二义性？

* `class Derived : [继承方式] Base1, [继承方式] Base2 { ... };`
* 若存在同名成员，需使用作用域限定符进行区分 `Base1::member` `Base2::member`

> 什么是菱形继承？存在什么缺点？C++ 如何解决“菱形继承”问题？

* 菱形继承：派生类从两个基类继承，而这两个基类又继承自同一个共同基类，形成菱形结构
* 缺点：共同基类的成员会被继承两次，导致**重复和数据冗余**
* 解决方法：使用虚继承，确保共同基类只被继承一次

> 简述 C++ 中继承与虚继承的差异。

* 在菱形继承的情况下，普通继承会多次继承同一共同基类
* 采用虚继承时，共同基类只会被继承一次，派生类共享同一基类子对象

```cpp
class A { public: A(){cout<<"A ";} };
class B : virtual public A { public: B(){cout<<"B ";} };
class C : virtual public A { public: C(){cout<<"C ";} };
class D : public B, public C { public: D(){cout<<"D ";} };
```

> 简述多继承会带来的问题。

* 名字冲突：多个基类中存在同名成员，导致二义性
* 耦合度高，内存布局复杂，增加复杂性，不便于维护

> 多继承情况下，派生类对象构造时，各部分的构造顺序是怎样的？内存上有什么差别？

* 构造顺序
  * 先构造虚基类
  * 再按声明顺序，依次构造非虚基类
  * 最后构造派生类成员和派生类本身
* 内存布局：虚基类只存在一份

> 在使用虚基类的情况下，虚基类的构造函数由谁负责调用？为什么要这样设计？

* 由最派生类负责调用虚基类的构造函数
* 设计原因：确保虚基类只被构造一次，避免重复构造

> 若虚基类只有参数化构造函数，最派生类应如何正确初始化虚基类？

* 在最派生类的初始化列表中，显式调用虚基类的参数化构造函数
* 中间类对虚基类的初始化会被忽略

## 多态

> 什么是多态？C++ 中有哪些实现多态的方式？

* 多态：同一操作作用于不同类型的对象时，可以表现出不同的行为
* 实现方式
  * 编译时：函数重载、运算符重载、模板
  * 运行时：虚函数机制

### 虚函数

> 编译器在哪些情况下，需要在同名函数中做选择？如何选择？

* 函数重载：在类中存在多个同名，但参数列表不同的函数，在编译时根据参数类型和数量选择函数
* 函数覆盖（重写）：派生类中存在与基类同名函数，参数列表相同，若基类函数为虚函数，则在运行时选择，否则在编译时静态绑定
* 注：若派生类中定义了与基类同名但参数列表不同的函数，**则基类同名函数会被隐藏**

> 静态绑定和动态绑定的区别？C++ 默认采用哪种绑定方式？为什么？

* 静态绑定：在编译时决定函数调用，效率高，但不支持多态
* 动态绑定：在运行时决定函数调用，支持多态，但效率较低
* 默认静态绑定：和 C 兼容，避免运行时开销

> 虚函数的作用是什么，有哪些性质？

* 作用：在运行时实现多态，根据对象的实际类型调用函数
* 性质：如基类中被定义为虚成员函数，则派生类中对其重定义的成员函数均为虚函数
  * 类的成员函数可以是虚函数
  * 构造函数不能是虚函数：在对象创建时，类型尚未确定，必须明确指定构造的类型
  * `static` 成员函数：`static` 函数属于类而非对象实例，无法实现动态绑定
  * 声明为 `inline` 的函数在动态绑定时无法内联展开
  * 析构函数往往是虚函数：通过基类指针删除派生类对象时，能正确调用派生类的析构函数，否则派生类持有的资源无法释放

> `override` `final` 关键字的作用？

* `override`：显式声明覆盖基类虚函数，只有基类存在虚函数才能通过编译检查，避免因参数列表写错将覆盖变成重载
* `final`
  * 函数：不能被重写
  * 类：不能被继承

> 虚函数是怎么实现多态的？

* 编译时创建虚函数表：存放指向虚函数的指针
* 对象内存布局最前方存放虚指针：指向虚函数表
* 运行时通过虚指针查找虚函数表，调用正确的函数实现

> 虚函数调用中，哪些部分由编译器决定，哪些部分由运行时决定？

* 编译器决定：是否为虚函数调用、默认参数、调用权限控制
* 运行时决定：具体调用哪个函数实现

> 什么是纯虚函数？抽象类的作用？

* 纯虚函数：在基类中声明但不定义的虚函数，使用 `= 0` 语法
* 抽象类：包含至少一个纯虚函数的类，不能实例化，只能作为基类，用作其他类的接口

> 为什么不应在派生类中重新定义与基类非虚成员函数同名的成员函数？这样做会带来什么问题？

* 会隐藏基类的同名函数，变成静态绑定，破坏预期的多态行为
* 如果需要实现多态，应该将基类函数声明为虚函数

> 比较纯虚函数、虚函数和非虚函数在接口继承、实现继承以及多态支持方面的差异。

| 函数类型           | 是否继承接口 | 是否继承实现  | 子类是否必须 override | 多态支持        |
| -------------- | ------ | ------- | --------------- | ----------- |
| 纯虚函数（`=0`）     | 是      | 否       | 是               | 是           |
| 虚函数（`virtual`） | 是      | 是（默认实现） | 否，若有额外需求可覆盖     | 是           |
| 非虚函数           | 是      | 是（固定实现） | 否，不应覆盖基类实现      | 否（静态绑定，非多态） |

### 运算符重载

> 什么是运算符重载？为什么要使用运算符重载？

* 运算符重载：为自定义类型定义运算符的行为
* 原因：使用自定义类型时，能像内置类型一样使用运算符，提高代码可读性和可维护性

> 为什么 C++ 规定“至少有一个操作数必须是自定义类型”才能重载运算符？

* 若所有操作数均为内置类型，编译器已有默认实现，重载会引起歧义

> 运算符重载有哪些性质？

* 不会破坏运算符的优先级和结合性

> 哪些运算符不能被重载？哪些操作符不能在类外重载？

* 不能重载：`::` `.` `.*` `->*` `sizeof` `typeid` `?:`
* 不能在类外重载：`=` `[]` `()` `->`

> 哪些操作符不应被重载？哪些推荐在类外重载？哪些推荐在类内重载？

* 不应重载：`&&` `||` 破坏短路求值
* 类外重载
  * 算术运算符：`+` `-` `*` `/` `%`
  * 流操作符：`<<` `>>`
* 类内重载
  * 无法在类外重载的操作符：`=` `[]` `()` `->`
  * 复合赋值运算符：`+=` `-=` `*=` `/=` `%=` 等

> 各类操作符应该返回什么类型？为什么？

* 算术运算符：返回新对象（值类型），避免修改原对象
* 赋值运算符：返回 `*this` 的引用，支持链式赋值
* 比较运算符：返回 `bool` 类型，表示比较结果

> 重载流操作符的函数签名是什么样的？为什么这样设计？

* `std::ostream& operator<<(std::ostream &os, const ClassName &obj);`
* 设计原因：返回 `ostream&` 以支持链式调用，参数为引用以避免拷贝开销

> 重载 `a++` 和 `++a` 有什么区别？如何区分？

|      | `a++`                       | `++a`                     |
| ---- | --------------------------- | ------------------------- |
| 函数签名 | `ClassName operator++(int)` | `ClassName& operator++()` |
| 操作   | 先返回旧值的副本，再增加                | 先增加，再返回自身引用               |
| 参数   | `int`（占位符，用于区分两种操作符）        | 无参数                       |

> 默认赋值函数有什么特点？`=` 操作符重载的签名是什么？

* 特点
  * 浅拷贝成员，递归调用成员类的赋值函数
  * 链式调用，返回赋值对象的引用
  * 不可继承
* 签名
  * 拷贝赋值：`ClassName& operator=(const ClassName &other);`
  * 移动赋值：`ClassName& operator=(ClassName &&other);`
* 注意：做自赋值检查

> `[]` 操作符如何重载？作用是什么

* 作用：实现下标访问功能，同时增加边界检查等额外逻辑
* 签名
  * 非常量版本：`Type& operator[](size_t index);`，可修改元素
  * 常量版本：`const Type& operator[](size_t index) const;`，只读访问，用于常量对象

> `()` 操作符如何重载？

* 函数对象：`ReturnType operator()(ParamTypes... params);`
* 自定义类型转换：`operator Type();`
  * 可能产生二义性
  * 使用 `explicit operator Type();` 防止隐式转换，只能调用 `static_cast<Type>(obj);` 显式转换

> `->` 操作符如何重载？重载的意义是什么？

* 作用：实现智能指针等类型的成员访问功能：允许自定义类型像指针一样访问成员，简化语法，提高可读性
* 签名：`ReturnType* operator->();`（重载时视为单目操作符，仅接受 `this` 作为参数）
* 若返回类型为指针，则按照指针的语义解引用并访问成员
* 若返回类型为类类型，则继续调用该类的 `operator->`

> C++ 中重载 `new` 和 `delete` 的主要目的是什么？

* 应用程序接管内存的释放和分配，减少内存碎片，提高性能

> 为何 `operator new` 和 `operator delete` 都是 `static` 成员函数？

* 对象的内存分配和释放不依赖于具体对象实例

> 默认的 `new` 和 `delete` 做了哪些操作

* `new`：调用 `operator new(sizeof(ClassName))` 分配内存，然后调用构造函数初始化对象
* `delete`：调用析构函数清理对象，然后调用 `operator delete(ptr)` 释放内存
* 若为 `new[]` 和 `delete[]`，则会额外处理数组大小信息

> 如何重载 `new` ？如何重载 `delete`？

* `new`
  * `static void* operator new(size_t size);`：重载全局 `new`，`size` 为分配的字节数，调用时编译器自动传入
  * `static void* operator new(size_t size, std::nothrow_t&) noexcept;`：不抛异常，失败返回 `nullptr`
  * `static void* operator new(size_t size, void* ptr) noexcept;`：Placement `new`，在 `ptr` 构造对象
* `delete`：`delete` 的重载唯一，重载后默认的版本失效
  * `static void operator delete(void* ptr);`：重载全局 `delete`，`ptr` 为要释放的内存地址
  * `static void operator delete(void* ptr, size_t size);`：带大小参数的版本
